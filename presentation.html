What the Data Can't Yet Tell Us

The analysis is only as good as the data it's built on. Several key dimensions are currently absent or ambiguous.

Completion vs dropout conflated For a 1-year programme, "not active next year" covers both successful completers and genuine dropouts. SGBSTDN_STST_CODE disambiguates — but graduate status codes need Admissions confirmation.
No socioeconomic signal Banner doesn't capture financial hardship, caring responsibilities, or employment hours — all documented dropout risk factors outside institutional view.
RSTS code semantics "Dropped" vs "Withdrawn" vs "Leave of Absence" may share code prefixes. Interpreting SFRSTCR_RSTS_CODE accurately requires Admissions confirmation of IADT's specific code set.
Annual term granularity One term code per academic year (YYYY00). Intra-year timing — October vs March withdrawal — is invisible at this resolution.

Module readable names require a JOIN to SCBCRSE. PSYL_H3010 title confirmed: "Intro to Cyberpsychology".

45 s. The most important limitation for this specific programme: we cannot yet distinguish students who completed from students who dropped out without additional status code analysis.

Next Steps & Conclusions

AdminPortal has proved the core proposition: centralised data access enables questions that weren't answerable before — in real time, against live data, for any programme.

01

Complete/dropout disambiguation — join SGBSTDN_STST_CODE to the inactive analysis to separate genuine dropouts from programme completers. This is the most urgent analytical gap.

02

Grade-to-dropout linkage — for students who became inactive, was there a grading signal in the same term? Query Q7 is already written to answer this.

03

Predictive flagging — combine grade trajectory with enrolment status to surface at-risk students before withdrawal, enabling earlier personal tutoring intervention.

04

Cross-programme patterns — the same analytical framework applies to any IADT programme. This is one example of a repeatable institution-scale analysis.

The Banner write pipeline already exists (Dataverse, EventMap push). Feeding intervention flags back into SIS via the Student Management module is a near-term possibility.

45 s. The hard infrastructure is done. The most urgent next step is actually simple: add a status-code join to distinguish completers from dropouts. Every IADT programme can be analysed the same way. Invite questions.
